# 采购领域知识图谱智能体 —— 技术方案

> 范围：架构设计 + 三个核心机制（Schema 建图 / 表间关系挖掘 / 图算法洞察）+ Neo4j 数据模型 + Agent 集成 + 落地计划。本文档为设计文档，不含完整代码实现，仅含必要的数据模型、流程与接口示意。

---

## 0. 设计前提与关键决策

| 决策项 | 选择 | 说明 |
| --- | --- | --- |
| 图数据库 | **Neo4j**（含 GDS 插件） | 字段级建图、领域子图标签、图算法 |
| Agent / Tool 框架 | **minimal-harness 0.6.1** | Registry + Factory + Middleware + AsyncIterator 事件流 |
| 底层业务库 | **PostgreSQL** | `information_schema` / `pg_catalog` 反向抽取元数据 |
| Schema 来源 | **自动抽取 + 人工补充** | 系统表自动建骨架，人工补领域标签与业务语义 |
| 关系挖掘 | **sqlglot 静态解析 + LLM 语义判断** | AST 抽 JOIN/外键/字段映射，LLM 判定业务语义 |
| 图算法 | **GDS（中心性/社区/多分辨率 Louvain）** | 全部库内跑，无需数据搬运 |

**核心原则（你提出的）**：
1. **只有 Schema 入图**，实例数据不入图，运行时通过 SQL 查库获取。
2. 图建到**字段级**，字段是一等公民节点。
3. Schema 支持**分领域打标签**，可按领域抽取子图查询。

## 1. 总体架构

### 1.1 分层架构

```
┌─────────────────────────────────────────────────────────────┐
│  L4  应用层：采购智能体（灵活查数/风险检查/血缘/洞察/运维）         │
│      基于 minimal-harness AgentRuntime + 多 Agent 编排           │
├─────────────────────────────────────────────────────────────┤
│  L3  工具层：minimal-harness Tool Registry                      │
│   schema_query / subgraph_fetch / lineage_trace /              │
│   sql_executor / risk_check / graph_insight ...                │
├─────────────────────────────────────────────────────────────┤
│  L2  知识图谱服务层                                              │
│   ① Schema 注册表 & 建图引擎   ② 关系挖掘引擎   ③ 图算法引擎       │
├─────────────────────────────────────────────────────────────┤
│  L1  存储层                                                     │
│   Neo4j(Schema图+GDS) │ Registry(PG/YAML) │ 业务库(PG，只读取数) │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 数据流总览

```
  [PG information_schema] ──抽取──┐
                                  ▼
  [人工领域标签/业务语义] ──→ ① Schema 注册表(SSOT) ──生成──→ [Neo4j Schema 图]
                                                                    ▲
  [几千个存储过程 .sql] ──→ ② 关系挖掘引擎(sqlglot+LLM) ──关系边──┘
                                                                    │
                           ③ 图算法引擎(GDS) ──中心性/社区/聚类──┘
                                                                    │
                                                                    ▼
                          L3 工具 ←── L4 采购智能体 ←── 用户
```

**关键点**：注册表是**单一可信源（SSOT, Single Source of Truth）**。Neo4j 图是注册表的**物化投影**，可随时由注册表幂等重建。关系挖掘与图算法的产物也回写注册表，保证图可重建、可审计。

## 2. Neo4j 数据模型（Schema 图）

### 2.1 节点类型（Node Labels）

| Label | 含义 | 关键属性 |
| --- | --- | --- |
| `Database` | 数据库/实例 | `name`, `engine=postgresql` |
| `Schema` | PG schema | `name` |
| `Table` | 表/视图 | `fqn`(全限定名), `name`, `type`(table/view), `row_estimate`, `comment` |
| `Column` | **字段（一等节点）** | `fqn`, `name`, `data_type`, `nullable`, `is_pk`, `is_fk`, `comment`, `semantic_type` |
| `Domain` | 业务领域 | `name`(如 采购/供应商/合同), `code` |
| `BusinessObject` | 业务对象（逻辑实体） | `name`(如 采购订单/供应商主数据), `code` |
| `Procedure` | 存储过程 | `name`, `file_path`, `loc`(行数) |

> 领域标签采用**双轨制**：既用 `Domain` 节点 + `:IN_DOMAIN` 关系（支持一表多领域、带权重），又在 `Table`/`Column` 上冗余一个 `domains: [string]` 数组属性（便于 Cypher 直接 `WHERE 'procurement' IN n.domains` 快速过滤）。两者由注册表统一生成，避免不一致。

### 2.2 关系类型（Relationship Types）

**结构关系（确定性，来自注册表）**

| 关系 | 方向 | 含义 |
| --- | --- | --- |
| `:HAS_SCHEMA` | Database→Schema | |
| `:HAS_TABLE` | Schema→Table | |
| `:HAS_COLUMN` | Table→Column | |
| `:REFERENCES` | Column→Column | 物理外键 |
| `:IN_DOMAIN` | Table/Column→Domain | 领域归属，属性 `weight`, `source`(manual/inferred) |
| `:REALIZES` | Table→BusinessObject | 物理表实现业务对象 |

**挖掘关系（推断性，来自机制②③，全部带 `confidence`/`source`/`evidence`）**

| 关系 | 方向 | 含义 | 关键属性 |
| --- | --- | --- | --- |
| `:JOINS_WITH` | Column↔Column | SQL 中频繁 JOIN | `frequency`, `join_type`, `confidence` |
| `:DERIVES_FROM` | Column→Column | 字段血缘（A 由 B 计算/赋值） | `transform`, `proc_refs`, `confidence` |
| `:RELATED_TO` | BusinessObject↔BusinessObject | LLM 判定的业务语义关系 | `semantic`(如 下单/履约), `confidence`, `evidence` |
| `:SIMILAR_TO` | Table↔Table / Column↔Column | 结构/命名/语义相似 | `score`, `method` |
| `:CO_USED_WITH` | Table↔Table | 同一存储过程共现 | `pmi`, `count` |

### 2.3 约束与索引（建图前置）

```cypher
CREATE CONSTRAINT table_fqn  IF NOT EXISTS FOR (t:Table)  REQUIRE t.fqn IS UNIQUE;
CREATE CONSTRAINT column_fqn IF NOT EXISTS FOR (c:Column) REQUIRE c.fqn IS UNIQUE;
CREATE CONSTRAINT domain_code IF NOT EXISTS FOR (d:Domain) REQUIRE d.code IS UNIQUE;
CREATE INDEX col_domains IF NOT EXISTS FOR (c:Column) ON (c.domains);
```

### 2.4 FQN —— 图谱全局主键

图中每个 Table/Column 节点用一个**全限定名（Fully Qualified Name）** 作为全局唯一 ID：

```text
fqn = {database}.{schema}.{table}[.{column}]
                    ↓                    ↓
              proc_db.public.po_order.supplier_id
                    ↑                    ↑
               表级别 FQN            字段级别 FQN
```

- **表级 fqn**：`proc_db.public.po_order` — 图内用作 `Table.fqn`
- **字段级 fqn**：`proc_db.public.po_order.supplier_id` — 图内用作 `Column.fqn`，也是 `reg_column` 的 PK
- **作用**：fqn 是跨注册表 ↔ 图库 ↔ 挖掘引擎三方对齐的唯一锚点。建图引擎做 `MERGE (c:Column {fqn: $fqn})` 保证幂等，关系挖掘的输出也用 fqn 表示起点和终点关系，再靠 fqn 写回注册表。

## 3. 机制① —— Schema 注册表 + 建图引擎

### 3.1 注册表设计（SSOT）

注册表是结构化的、可增删改、可版本化的元数据库。建议**主存 PostgreSQL 表 + 导出 YAML 快照供 Git 审计**双形态。核心表：

**`reg_table`（表注册）**

| 字段 | 说明 |
| --- | --- |
| `fqn` (PK) | 全限定名 |
| `schema_name`, `table_name` | |
| `type` | table/view |
| `business_object` | 关联业务对象 code |
| `domains` | 领域 code 数组（人工/推断） |
| `comment` | 业务描述 |
| `status` | active/deprecated |
| `source` | introspect/manual |
| `version`, `updated_at`, `updated_by` | 变更审计 |

**`reg_column`（字段注册）**

| 字段 | 说明 |
| --- | --- |
| `fqn` (PK) | |
| `table_fqn` (FK) | |
| `name`, `data_type`, `nullable`, `ordinal` | |
| `is_pk`, `is_fk`, `ref_column_fqn` | 主外键 |
| `semantic_type` | 业务语义类型（如 金额/数量/供应商编码/状态码） |
| `domains`, `comment`, `tags` | |
| `status`, `source`, `version` | |

**`reg_domain`（领域字典）**：`code`, `name`, `parent_code`(支持领域树), `description`。

**`reg_relationship`（关系注册表）**

> 机制②③ 挖掘出的关系、以及手工录入的业务语义关系，统一先写这张表，再由建图引擎投影到 Neo4j。**所有关系类型的全文持 SSOT 纪律——关系入注册表，不入注册表的关系不入图。**

| 字段 | 说明 |
| --- | --- |
| `id` (PK) | 自增/BIGINT，纯代理主键，无业务含义 |
| `src_fqn` (FK→reg_table.fqn 或 reg_column.fqn) | 起点：字段级关系指向 `reg_column.fqn`，表级关系指向 `reg_table.fqn` |
| `dst_fqn` (FK→reg_table.fqn 或 reg_column.fqn) | 终点，同上 |
| `node_level` | `column` / `table` / `business_object` — 区分关系在哪一层级 |
| `rel_type` | `REFERENCES` / `JOINS_WITH` / `DERIVES_FROM` / `SIMILAR_TO` / `CO_USED_WITH` / `RELATED_TO` |
| `is_directed` | boolean — 真则表示有方向（如 DERIVES_FROM），假则表示双向（如 JOINS_WITH） |
| `properties` | JSONB — 关系属性：`{frequency: 5, confidence: 0.85, source: "inferred:sqlglot", evidence: ["SELECT ... JOIN ..."], join_type: "LEFT", pmi: 0.3}` |
| `source` | `introspect`(来自 PG 外键) / `inferred:sqlglot` / `inferred:llm` / `manual` |
| `status` | `active` / `deprecated` |
| `version`, `created_at`, `updated_at` | 版本和审计 |

**PK vs FK 速览**：

| 表 | PK | 说明 |
| --- | --- | --- |
| `reg_table` | `fqn` (自然键) | 全限定名本身就是唯一的业务主键 |
| `reg_column` | `fqn` (自然键) | 同上 |
| `reg_relationship` | `id` (代理键) | 自增整数，纯粹的物理行标识。`src_fqn`/`dst_fqn` 是 FK 指向 `reg_column.fqn` 或 `reg_table.fqn` |

> 关系的 `rel_type` + `src_fqn` + `dst_fqn` 组合有唯一约束，避免重复行。`id` 仅仅为了索引效率和变更日志引用。

**字段级 vs 表级关系的区分方式**：

```text
字段级（常态）：                        表级（高层次）：
src_fqn = reg_column.fqn               src_fqn = reg_table.fqn
dst_fqn = reg_column.fqn               dst_fqn = reg_table.fqn
node_level = 'column'                  node_level = 'table'
```

字段级和表级关系在同一个 `reg_relationship` 表中，通过 `node_level` 字段区分。建图引擎按 `node_level` 决定 MERGE 到 Column 级别还是 Table 级别。

**`reg_change_log`（变更流水）**：记录 `reg_table` / `reg_column` / `reg_relationship` 每次 upsert/delete 的 diff，支撑增量建图与回滚。

### 3.2 三阶段填充

```
阶段A 自动反向抽取（introspect）
  从 PG 读取：
    information_schema.tables / columns
    table_constraints / key_column_usage / referential_constraints (外键)
    pg_description (表/列注释)
  → 写入 reg_table / reg_column，source=introspect
  → semantic_type 用规则+LLM 初判（如列名含 amt/amount→金额）

阶段B 人工领域标定与语义补充（manual override）
  领域专家在管理界面/YAML 中：
    - 给表/字段打 domains 标签
    - 关联 business_object
    - 修正 semantic_type、补 comment
  → source=manual 的字段优先级高于 introspect，重抽取不覆盖人工值

阶段C 挖掘回填（inferred，来自机制②③）
  关系/相似度/领域推断结果回写注册表，source=inferred，带 confidence
```

> **冲突合并策略**：同一字段属性优先级 `manual > inferred > introspect`。重新抽取只更新 introspect 层，人工层不被覆盖（分层存储 + 视图合并）。

### 3.3 建图引擎（注册表 → Neo4j）

**幂等 MERGE 投影**，支持全量重建与增量同步：

```cypher
// 表节点
MERGE (t:Table {fqn:$fqn})
  SET t.name=$name, t.type=$type, t.comment=$comment,
      t.domains=$domains, t.status=$status, t.version=$version
// 字段节点 + 归属
MERGE (c:Column {fqn:$col_fqn})
  SET c += $col_props
MERGE (t)-[:HAS_COLUMN]->(c)
// 领域标签（双轨：节点关系 + 数组属性）
WITH c UNWIND $domains AS dc
  MATCH (d:Domain {code:dc})
  MERGE (c)-[r:IN_DOMAIN]->(d) SET r.source=$src
  // 外键（物理外键来自 introspect）
MATCH (a:Column {fqn:$col_fqn}), (b:Column {fqn:$ref_fqn})
  MERGE (a)-[:REFERENCES]->(b)

// 挖掘关系（来自 reg_relationship，带属性）
MATCH (src:Column {fqn:$src_fqn}), (dst:Column {fqn:$dst_fqn})
  CALL apoc.create.relationship(src, $rel_type, $rel_props, dst) YIELD rel
  RETURN rel
```

**增量同步**：监听 `reg_change_log`，按 diff 只 MERGE/DELETE 受影响子图，避免全量重跑。

**增删改支持**：
- **增**：新表/字段 → 新 MERGE。
- **改**：属性变更 → SET 覆盖（version+1）。
- **删**：注册表标 `status=deprecated`（软删），图中保留节点但打标；硬删走显式 `DETACH DELETE` 并记日志。

### 3.4 领域子图抽取（满足"按领域查询"）

```cypher
// 抽取"采购"领域子图
MATCH (d:Domain {code:'procurement'})
MATCH (c:Column)-[:IN_DOMAIN]->(d)
MATCH (t:Table)-[:HAS_COLUMN]->(c)
OPTIONAL MATCH (c)-[r:REFERENCES|JOINS_WITH|DERIVES_FROM]-(c2:Column)
RETURN t, c, r, c2
```

或用 GDS 子图投影 `gds.graph.project.cypher` 把领域子图投到内存图，供算法或快速查询复用。

## 4. 机制② —— 表间关系挖掘（sqlglot + LLM）

### 4.1 总体流程

```
几千个 .sql 文件
   │ ① 分片与预处理（按 proc 切分、去注释、参数归一）
   ▼
sqlglot AST 解析（PostgreSQL 方言）
   │ ② 静态抽取（确定性证据）
   │   - JOIN ... ON a.x = b.y      → 候选 JOINS_WITH
   │   - INSERT/UPDATE ... SELECT    → 字段血缘 DERIVES_FROM
   │   - FROM 多表共现               → CO_USED_WITH
   │   - 子查询/CTE 依赖链           → 传递血缘
   ▼
证据聚合（跨文件汇总：frequency / PMI / 共现矩阵）
   │ ③ LLM 语义判断（仅对高价值/歧义候选）
   │   - 给定两表+列+采样 SQL 片段，判定业务语义关系名与方向
   │   - 输出 RELATED_TO(semantic, confidence, evidence)
   ▼
关系入注册表(source=inferred) → 建图引擎回写 Neo4j
```

### 4.2 静态解析层（sqlglot，确定性）

对每个存储过程：

1. **解析 AST**：`sqlglot.parse(sql, read='postgres')`，得到语句序列。
2. **JOIN 关系**：遍历 `Join` 节点，提取 `ON` 谓词中的 `a.col = b.col`，解析列归属到 `fqn`。统计跨全库的 `(colA, colB)` 共现 `frequency` 与 `join_type`。
3. **字段血缘**：
   - `INSERT INTO t1(c1) SELECT expr(c2) FROM t2` → `t1.c1 :DERIVES_FROM t2.c2`，记录 `transform`(表达式文本)。
   - `UPDATE ... SET c1 = expr` 同理。
   - CTE/子查询：构建别名→真实表的解析作用域（scope resolution），逐层回溯到物理列。
4. **表共现**：同一 proc 的 `FROM`/`JOIN` 表集合，两两累加 `CO_USED_WITH.count`，全局算 **PMI**（点互信息）衡量非偶然共现强度。
5. **列名解析兜底**：未带表前缀的列，用注册表 `reg_column` 反查唯一归属；歧义列标记待 LLM/人工裁决。

**输出**：结构化候选关系表 `mining_candidate(src_fqn, dst_fqn, rel_type, evidence_sql[], frequency, confidence_static)`。

### 4.3 LLM 语义判断层（minimal-harness Tool）

**不对几千文件全量调 LLM**（成本/幻觉风险）。仅在以下场景触发：

- 静态证据强但**缺业务语义命名**的高频 JOIN → 让 LLM 命名业务关系（如 `po_order.supplier_id ↔ supplier.id` → 语义 `下单于`）。
- **歧义列归属**裁决。
- **业务对象级关系**（`BusinessObject` 间 `RELATED_TO`）归纳。

LLM 调用封装为 minimal-harness 工具，Prompt 模式：

```
输入：表A/表B 的字段清单+注释、Top-K 条真实 JOIN SQL 片段、领域上下文
要求：输出 JSON
  { "semantic": "下单关系", "direction": "A->B",
    "confidence": 0.0-1.0, "evidence": "...", "abstain": false }
约束：证据不足必须 abstain=true，禁止臆造
```

**质量护栏**：
- 所有 LLM 关系 `confidence < 阈值` 进**人工复核队列**，不直接入图。
- `evidence` 字段强制保留可溯源 SQL 片段。
- 用 minimal-harness `Middleware` 记录每次 LLM 调用 token 与裁决日志。

### 4.4 规模化与增量

- **并行分片**：几千文件按目录/模块分批，sqlglot 解析天然可并行（多进程）。
- **失败隔离**：单文件解析失败（方言/动态 SQL 拼接）记入 `parse_error` 表，不阻断整体；动态 SQL 降级为正则提取表名兜底。
- **增量挖掘**：按文件 hash 缓存解析结果，仅重解析变更文件。

## 5. 机制③ —— 图算法洞察（GDS 全量库内计算）

所有图算法均在 Neo4j GDS 库内完成，无需数据搬运。结果统一通过 `gds.*.write` 回写节点属性，同时回填 `reg_relationship`（关系级结果）或 `reg_column`/`reg_table`（属性级结果），再经建图引擎统一物化到图。

### 5.1 算子矩阵

| 分析目标 | GDS 算子 | 业务解读 | 回写目标 |
| --- | --- | --- | --- |
| 关键表/字段识别 | `degree` / `pageRank` / `betweenness` | 高中心性=数据枢纽/血缘咽喉，治理重点 | `Table.pagerank`, `Column.betweenness` |
| 业务模块发现 | `louvain`(多分辨率) / `leiden` | 自动聚成业务模块，校验/补充领域标签 | `Table.community_id` 各级 |
| 强连通分量 | `wcc`（弱连通分量） | 发现孤岛表/未连通子系统 | `Table.wcc_id` |
| 字段结构相似 | `nodeSimilarity`（Jaccard，基于邻居拓扑） | 结构相似字段→主数据/冗余识别 | `:SIMILAR_TO` 写入 `reg_relationship` |
| 血缘影响面 | `bfs`/`dfs` from 节点 | 改一个字段影响哪些下游 | 运行时查询，不入持久化 |

### 5.2 Louvain 多分辨率 —— 替代层次聚类

GDS Louvain 原生支持 `includeIntermediateCommunities: true`，在一次运行中输出**每一轮聚合的中间社区 ID**，天然形成层次树，等价于层次聚类的多粒度产出：

```cypher
CALL gds.graph.project('proc_kg',
  ['Table','Column'],
  {HAS_COLUMN:{}, REFERENCES:{}, JOINS_WITH:{properties:'frequency'}});

// 多分辨率 Louvain：一次跑出各级社区
CALL gds.louvain.write('proc_kg', {
  writeProperty: 'community_id',         // 最终（最粗粒度）社区
  includeIntermediateCommunities: true,   // 输出每轮中间结果
  maxLevels: 10
}) YIELD communityCount, modularity, ranLevels;

// 如果粗粒度社区太粗糙，调 resolution 参数获得更细划分
CALL gds.louvain.write('proc_kg', {
  writeProperty: 'community_fine',
  resolution: 1.5,                        // >1 产生更细社区
  includeIntermediateCommunities: true
});
```

结果 `community_id` 是多值数组：`[round1_id, round2_id, ..., final_id]`，从细到粗对应业务子模块→业务域。可直接反哺 `reg_domain` 的领域树层级建议。

**为什么不需要 Python 层次聚类：**

| 对比项 | GDS Louvain 多分辨率 | Python (scipy) 层次聚类 |
| --- | --- | --- |
| 数据搬运 | 零，图库内直接算 | 需导出整图到 Python，大图搬运慢 |
| 多粒度 | `includeIntermediateCommunities` 一次输出 | 自己切 dendrogram |
| 扩展性 | 亿级节点 | 受限于 Python 内存 |
| 维护 | GDS 原生，零额外依赖 | networkx+scipy 流水线 |
| 分辨率调节 | `resolution` 参数任意调 | 换 linkage 方法重跑 |

### 5.3 GDS 结果回写注册表流程

```text
GDS write 到 Neo4j 节点属性
    │
    ▼
Python 轻量脚本（可选）：
  - 读取 Neo4j 中的算法结果
  - 与人工领域标签做一致性比对（如：Louivain 社区与人工 Domain 标签的 purity/NMI）
  - 产出比对报告，标记高差异节点供人工复核
    │
    ▼
算法结果（如社区ID、PageRank 值）作为 inferred 层回填到 reg_table / reg_column 属性字段
  → 建图引擎下次重建时自动同步

关系级算法结果（如 nodeSimilarity）回填到 reg_relationship，source=inferred
```

> 所有算子运行记录带 `algo_version` / `run_at` / `parameters`，写入 `reg_algo_log` 表供审计与可复现。

## 6. Agent / Tool 层（minimal-harness 集成）

### 6.1 工具注册（ToolRegistry）

每个能力封装为 minimal-harness 工具（`ToolMetadata + LocalToolBinding`，async generator）：

| 工具 | 功能 | 底层 |
| --- | --- | --- |
| `schema_search` | 按关键词/领域检索表/字段 | Neo4j 查询 |
| `subgraph_fetch` | 抽取指定领域/业务对象子图 | Cypher / GDS 投影 |
| `lineage_trace` | 字段血缘上下游追溯 | `:DERIVES_FROM` BFS |
| `join_path_find` | 两表间可达 JOIN 路径（自动拼查询） | `:JOINS_WITH` 最短路 |
| `sql_executor` | 拿到 Schema 后真正查业务库取实例数据 | PG 只读连接 |
| `risk_check` | 风险规则校验（如关键字段无外键约束、跨域引用） | 图模式匹配 |
| `graph_insight` | 返回中心性/社区/聚类洞察 | 读 GDS 回写属性 |

> 呼应你的核心设计：**Schema 在图里，实例数据靠 `sql_executor` 实时查库**。Agent 先用图工具定位"查什么表哪些字段、怎么 JOIN"，再用 `sql_executor` 取数。

### 6.2 多 Agent 编排（AgentRuntime）

按你的五大场景注册专用 Agent（`AgentMetadata`，绑定不同工具子集与 system_prompt）：

| Agent | 场景 | 主用工具 |
| --- | --- | --- |
| `query_agent` | 灵活查数 | schema_search, join_path_find, sql_executor |
| `risk_agent` | 风险检查 | risk_check, lineage_trace |
| `lineage_agent` | 数据血缘挖掘 | lineage_trace, subgraph_fetch |
| `insight_agent` | 业务洞察 | graph_insight, subgraph_fetch |
| `ops_agent` | 运营运维 | schema_search, risk_check |

由一个 **router/orchestrator agent** 按用户意图分发，复用 `AgentRuntime` + `DiskSessionStore` 维持会话记忆。

### 6.3 治理护栏（Middleware）

- `should_allow_tool`：限制 `sql_executor` 只读、禁 DDL/DML、领域越权拦截。
- `on_llm_end`：记录 token 成本。
- 关系挖掘 LLM 裁决全程留痕，低置信结果不自动入图。

## 7. 落地计划（分阶段）

| 阶段 | 目标 | 关键交付 |
| --- | --- | --- |
| **P0 地基** | 注册表 schema + Neo4j 约束 + introspect 抽取 | 自动建出表/字段/外键骨架图 |
| **P1 领域标定** | 人工领域标签 + 业务对象 + 子图抽取 | 可按领域抽子图查询 |
| **P2 关系挖掘** | sqlglot 静态解析全量 SQL + 证据聚合 | JOINS_WITH / DERIVES_FROM / CO_USED_WITH 入图 |
| **P3 语义增强** | LLM 语义判断 + 人工复核队列 | RELATED_TO 业务关系 + 字段血缘 |
| **P4 图洞察** | GDS 中心性/社区/多分辨率 Louvain + 一致性比对 | 关键节点、领域树建议、一致性报告 |
| **P5 Agent 化** | minimal-harness 工具 + 多 Agent 编排 | 五大场景智能体上线 |

**贯穿全程**：注册表为 SSOT，图为可重建投影；所有推断结果带 `confidence/source/evidence`；变更走 `change_log` 支持增量与回滚。

## 8. 关键风险与对策

| 风险 | 对策 |
| --- | --- |
| 动态拼接 SQL / 非标准方言 sqlglot 解析失败 | 失败隔离 + 正则兜底提表名 + parse_error 复核队列 |
| LLM 关系幻觉 | 仅高价值候选调用 + 强制 evidence + 置信阈值 + 人工复核 |
| 人工标签与自动抽取冲突 | 分层存储，优先级 manual>inferred>introspect，重抽取不覆盖人工 |
| 图与注册表不一致 | 图为投影，幂等重建；change_log 驱动增量同步 |
| 几千文件解析性能 | 多进程分片 + 文件 hash 增量缓存 |
| 字段级图规模膨胀 | 领域子图投影按需加载；冷字段软删/归档 |
| GDS 图投影内存不足 | 只投对应领域子图 + Cypher 投影过滤；增大 Neo4j heap |
