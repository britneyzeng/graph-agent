# 采购领域知识图谱智能体 —— 技术方案（v2，当前实现）

> 范围：架构设计 + 三个核心机制（Schema 建图 / SQL 关系挖掘 / 图算法洞察）+ Kuzu 数据模型 + Agent 集成 + 落地计划。本文档反映当前已在代码库中实现的真实状态。

---

## 0. 设计前提与关键决策

| 决策项 | 选择（当前实现） | 说明 |
| --- | --- | --- |
| 图数据库 | **Kuzu**（嵌入式 C++ 图数据库，列式压缩存储） | 字段级建图、领域子图标签、图算法 |
| Agent / Tool 框架 | **minimal-harness** + mh-service-kit | Registry + Factory + Middleware + AsyncIterator 事件流 |
| 底层业务库 | **PostgreSQL** | `sql_executor` 只读查询 |
| Schema 注册表 | **Excel 文件**（openpyxl）**SSOT** | `manual_registry.xlsx`（5 个 Sheet），非 PG 表 |
| 关系挖掘 | **sqlglot 静态 AST 解析**，无 LLM | 提取 JOIN / 字段血缘 / 表共现 → FIELD_LINK + ENTITY_LINK |
| 图算法 | **NetworkX（Python）** | 从 Kuzu 导出子图到 NetworkX 内存，跑完回写；非 GDS |
| 关系命名 | **无 LLM 语义命名** | 所有关系仅 `source + status` 属性，无 confidence/frequency |

**核心原则（设计不变）**：
1. **只有 Schema 入图**，实例数据不入图，运行时通过 SQL 查库获取。
2. 图建到**字段级**，字段是一等公民节点。
3. Schema 支持**分领域打标签**，可按领域抽取子图查询。

## 1. 总体架构

### 1.1 分层架构

```
┌─────────────────────────────────────────────────────────────┐
│  L4  应用层：采购智能体（灵活查数/风险检查/血缘/洞察/运维）         │
│      基于 minimal-harness AgentRuntime + 多 Agent 编排           │
├─────────────────────────────────────────────────────────────┤
│  L3  工具层：minimal-harness Tool Registry (10个工具)            │
│   schema_search / subgraph_fetch / lineage_trace /              │
│   join_path_find / sql_executor / query_schema_* /              │
│   risk_check / graph_insight                                    │
├─────────────────────────────────────────────────────────────┤
│  L2  知识图谱服务层                                              │
│   ① Schema 注册表 & Kuzu 建图引擎   ② 关系挖掘引擎   ③ 图算法引擎   │
├─────────────────────────────────────────────────────────────┤
│  L1  存储层                                                     │
│   Kuzu(Schema图) │ Registry(manual_registry.xlsx) │ 业务库(PG)   │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 数据流总览

```
  [manual_registry.xlsx] ──加载──→ ① Schema 注册表(SSOT) ──生成──→ [Kuzu Schema 图]

  [SQL 文件] ─────────────────→ ② 关系挖掘(sqlglot) ──FIELD_LINK / ENTITY_LINK──→ [Kuzu 图]

  [Kuzu 图] ──导出 Cypher 结果──→ ③ NetworkX 算法 ──回写节点属性/FIELD_LINK──→ [Kuzu 图]

                                                            L3 工具 ←── L4 智能体 ←── 用户
```

**关键点**：`manual_registry.xlsx` 是**单一可信源（SSOT）**。Kuzu 图是注册表的**物化投影**，可随时由注册表幂等重建。关系挖掘与图算法的产物也回写图（FIELD_LINK / ENTITY_LINK / 节点属性），但**不反写 Excel**。

## 2. Kuzu 数据模型（Schema 图）

### 2.1 节点类型（Node Tables）

| Label | PK | 关键属性 |
| --- | --- | --- |
| `Domain` | `fqn` | name_cn, name_en, parent_fqn, description, source, status |
| `Entity` | `fqn` | entity_type, name_cn, name_en, src_tables[], domains[], description, source, status, property_count, pk_properties[], fk_properties[], has_pk, has_fk, properties, pagerank, betweenness, degree, community_id, wcc_id |
| `Field` | `fqn` | entity_fqn, name, data_type, is_pk, ref_property_fqn, description, name_cn, name_en, source, status, domains[], pagerank, betweenness, degree, community_id, wcc_id |
| `Logic` | `fqn` | logic_type, expression, name_cn, name_en, description, source, status |

FQN 格式：`{domain}.{schema}.{entity}[.{field}]`
- 实体级：`core.public.supplier`
- 字段级：`core.public.supplier.supplier_id`

### 2.2 关系类型（Relationship Tables）

共 9 张关系表，**全部仅含 `source, status` 两个属性**，无 confidence/frequency/evidence。

| Rel Table | FROM → TO | 含义 | 注册方式 |
| --- | --- | --- | --- |
| `IN_DOMAIN` | Entity → Domain | entity 归属 domain | 自动同步自 Entity.domains 数组 |
| `HAS_PROPERTY` | Entity → Field | entity 拥有字段 | 自动同步自 Property.entity_fqn |
| `COMPUTES` | Logic → Field | logic 计算 field 值 | Relationship Sheet（手动录入） |
| `DECOMPOSES_TO` | Logic → Logic | logic 分解为子 logic | Relationship Sheet（手动录入） |
| `FIELD_LINK` | Field → Field | FK 引用 / 挖掘的 JOIN 或血缘 | 自动同步自 ref_property_fqn + 挖掘产物 |
| `ENTITY_LINK` | Entity → Entity | 表共现关系 | 挖掘产物（co-occurrence） |
| `DOMAIN_LINK` | Domain → Domain | 父子 domain 层级 | 自动同步自 Domain.parent_fqn |
| `USE_LOGIC` | Entity → Logic | entity 使用 logic | Relationship Sheet（手动录入） |
| `HAS_LOGIC` | Domain → Logic | domain 包含 logic | Relationship Sheet（手动录入） |

### 2.3 Kuzu DDL

```cypher
CREATE NODE TABLE IF NOT EXISTS Domain(
    fqn STRING, name_cn STRING, name_en STRING,
    parent_fqn STRING, description STRING, source STRING, status STRING,
    PRIMARY KEY (fqn)
);

CREATE NODE TABLE IF NOT EXISTS Entity(
    fqn STRING, entity_type STRING, name_cn STRING, name_en STRING,
    src_tables STRING[], domains STRING[], description STRING,
    source STRING, status STRING, property_count INT64,
    pk_properties STRING[], fk_properties STRING[],
    has_pk BOOLEAN, has_fk BOOLEAN, properties STRING,
    pagerank DOUBLE, betweenness DOUBLE, degree DOUBLE,
    community_id INT64, wcc_id INT64,
    PRIMARY KEY (fqn)
);

CREATE NODE TABLE IF NOT EXISTS Field(
    fqn STRING, entity_fqn STRING, name STRING, data_type STRING,
    is_pk BOOLEAN, ref_property_fqn STRING,
    description STRING, name_cn STRING, name_en STRING,
    source STRING, status STRING, domains STRING[],
    pagerank DOUBLE, betweenness DOUBLE, degree DOUBLE,
    community_id INT64, wcc_id INT64,
    PRIMARY KEY (fqn)
);

CREATE NODE TABLE IF NOT EXISTS Logic(
    fqn STRING, logic_type STRING, expression STRING,
    name_cn STRING, name_en STRING, description STRING,
    source STRING, status STRING,
    PRIMARY KEY (fqn)
);

-- 9 张关系表
CREATE REL TABLE IF NOT EXISTS IN_DOMAIN(FROM Entity TO Domain, source STRING, status STRING);
CREATE REL TABLE IF NOT EXISTS HAS_PROPERTY(FROM Entity TO Field, source STRING, status STRING);
CREATE REL TABLE IF NOT EXISTS COMPUTES(FROM Logic TO Field, source STRING, status STRING);
CREATE REL TABLE IF NOT EXISTS DECOMPOSES_TO(FROM Logic TO Logic, source STRING, status STRING);
CREATE REL TABLE IF NOT EXISTS FIELD_LINK(FROM Field TO Field, source STRING, status STRING);
CREATE REL TABLE IF NOT EXISTS ENTITY_LINK(FROM Entity TO Entity, source STRING, status STRING);
CREATE REL TABLE IF NOT EXISTS DOMAIN_LINK(FROM Domain TO Domain, source STRING, status STRING);
CREATE REL TABLE IF NOT EXISTS USE_LOGIC(FROM Entity TO Logic, source STRING, status STRING);
CREATE REL TABLE IF NOT EXISTS HAS_LOGIC(FROM Domain TO Logic, source STRING, status STRING);
```

## 3. 机制① —— Schema 注册表 + 建图引擎

### 3.1 注册表设计（SSOT）— Excel 文件

注册表为 `registry/manual_registry.xlsx`，包含 5 个 Sheet：

**Sheet: Domain**

| 列 | 说明 |
| --- | --- |
| `fqn` (PK) | 领域代码，如 `core`, `procurement` |
| `name_cn` | 中文名 |
| `name_en` | 英文名 |
| `parent_fqn` | 父领域 fqn，空为根领域 |
| `description` | 描述 |
| `source` | manual |
| `status` | active / deprecated |

**Sheet: Entity**

| 列 | 说明 |
| --- | --- |
| `fqn` (PK) | `{domain}.{schema}.{entity}` |
| `entity_type` | 实体类型，如 `supplier`, `purchase_order` |
| `name_cn` | 中文名 |
| `name_en` | 英文名 |
| `src_tables` | JSON 数组，映射的物理表 |
| `domains` | JSON 数组，所属领域列表 |
| `description` | 描述 |
| `source` | manual |
| `status` | active / deprecated |

**Sheet: Property**

| 列 | 说明 |
| --- | --- |
| `fqn` (PK) | `{entity_fqn}.{field_name}` |
| `entity_fqn` | 所属 Entity fqn |
| `data_type` | 数据类型（string, int, decimal 等） |
| `is_pk` | 是否主键 |
| `ref_property_fqn` | 外键引用目标字段 fqn，不为空即表示 FK |
| `description` | 描述 |
| `name_cn` | 中文名 |
| `name_en` | 英文名 |
| `source` | manual |
| `status` | active / deprecated |

> **没有 `is_fk` 字段**：FK 状态通过 `ref_property_fqn` 非空判定。

**Sheet: Logic**

| 列 | 说明 |
| --- | --- |
| `fqn` (PK) | `{domain}.{logic_name}` |
| `logic_type` | 逻辑类型，如 `formula`, `rule`, `aggregation` |
| `expression` | 逻辑表达式 |
| `name_cn` | 中文名 |
| `name_en` | 英文名 |
| `description` | 描述 |
| `source` | manual |
| `status` | active / deprecated |

**Sheet: Relationship**

| 列 | 说明 |
| --- | --- |
| `src_fqn` | 起点 fqn |
| `dst_fqn` | 终点 fqn |
| `rel_type` | 关系类型，仅 4 种有效值：`COMPUTES`, `DECOMPOSES_TO`, `USE_LOGIC`, `HAS_LOGIC` |
| `source` | manual |
| `status` | active / deprecated |

> 其他 5 种关系（IN_DOMAIN, HAS_PROPERTY, FIELD_LINK, ENTITY_LINK, DOMAIN_LINK）为**自动同步**，不在 Relationship Sheet 中手工录入。

### 3.2 建图引擎（Excel → Kuzu）

`builder/graph_builder.py` 中的 `GraphBuilder` 类负责从 `RegistryData` 同步到 Kuzu。

**同步顺序**：
1. `_sync_domains()` — MERGE Domain 节点
2. `_sync_entities()` — MERGE Entity 节点 + IN_DOMAIN 关系（来自 Entity.domains）
3. `_sync_properties()` — MERGE Field 节点 + HAS_PROPERTY 关系（来自 Property.entity_fqn）
4. `_sync_logics()` — MERGE Logic 节点
5. `_enrich_entities()` — 补充 property_count / pk_properties / fk_properties 等派生属性
6. `_sync_domain_links()` — 根据 Domain.parent_fqn 创建 DOMAIN_LINK
7. `_sync_fk_field_links()` — 根据 Property.ref_property_fqn 创建 FIELD_LINK
8. `_sync_computes()` — 处理 COMPUTES 关系
9. `_sync_decomposes_to()` — 处理 DECOMPOSES_TO 关系
10. `_sync_relationships()` — 处理 USE_LOGIC / HAS_LOGIC 等剩余关系

**命令行**：`python -m scripts.sync_to_graph --xlsx registry/manual_registry.xlsx`

**幂等性**：每次全量重建（`get_kuzu_client(recreate=True)` 重建数据库）。

### 3.3 领域子图抽取

```cypher
// 抽取"采购"领域子图
MATCH (d:Domain {fqn: 'procurement'})
MATCH (e:Entity)-[:IN_DOMAIN]->(d)
MATCH (e)-[:HAS_PROPERTY]->(f:Field)
OPTIONAL MATCH (f)-[r:FIELD_LINK]-(f2:Field)
RETURN e, f, r, f2
```

## 4. 机制② —— 表间关系挖掘（sqlglot 静态解析）

### 4.1 总体流程

```
SQL 文件（存储过程 / 视图）
   │ ① sqlglot AST 解析（PostgreSQL 方言）
   ▼
三种证据提取（确定性，无 LLM）：
   - JOIN ... ON a.x = b.y  → 列←→列共现
   - INSERT/UPDATE ... SELECT → 字段血缘
   - FROM/JOIN 多表共现 → 表对共现
   │ ② 跨文件聚合（频率 / PMI）
   ▼
聚合结果写 Kuzu 图：
   - JOIN + 血缘 → FIELD_LINK（Field→Field）
   - 表共现 → ENTITY_LINK（Entity→Entity）
```

### 4.2 静态解析层（sqlglot）

`mining/sql_parser.py` — `SqlParser` 类，提取 3 类证据：

| 证据 | 来源 SQL 模式 | 输出结构 | 聚合后目标关系 |
| --- | --- | --- | --- |
| JOIN | `SELECT ... FROM t1 JOIN t2 ON t1.a = t2.b` | `JoinEvidence(src_col_fqn, dst_col_fqn, join_type, context_sql)` | FIELD_LINK |
| Lineage | `INSERT INTO t1(c1) SELECT expr(c2) FROM t2` / `UPDATE t1 SET c1 = expr` | `LineageEvidence(src_col_fqn, dst_col_fqn, transform, context_sql)` | FIELD_LINK |
| Co-occurrence | 同一 SQL 中的 `FROM` / `JOIN` 表集合 | `CooccurrenceEvidence(table_fqns)` | ENTITY_LINK |

**关键实现细节**：
- 列名解析：通过 `Scope` 类处理表别名→物理名映射
- 等值条件提取：`_extract_equalities` 处理 EQ 表达式，含 Cast 包裹的列
- 血缘提取：INSERT 语句的 `_extract_lineage_insert` 和 UPDATE 语句的 `_extract_lineage_update`
- 解析失败兜底：`regex_extract_tables` 正则提取表名

### 4.3 聚合层

`mining/relation_aggregator.py` — `RelationAggregator` 类：

| 聚合器 | 数据结构 | 输出 |
| --- | --- | --- |
| JOIN 聚合 | `joins: dict[(src_fqn,dst_fqn), AggregatedJoin]` | `get_joins(min_freq=2)` → FIELD_LINK |
| Lineage 聚合 | `lineages: dict[(src_fqn,dst_fqn), AggregatedLineage]` | `get_lineage()` → FIELD_LINK |
| 表共现 | `cooccurrences: dict[(tableA,tableB), AggregatedCooccurrence]` | `get_cooccurrences(min_pmi=0.0)` → ENTITY_LINK |

- JOIN 按 `_ordered_key` 去重 + 统计 `frequency`
- 表共现计算 **PMI**（点互信息）：`pmi = log2(P(a,b) / (P(a) * P(b)))`
- 每类聚合结果输出为 `{"src_fqn": ..., "dst_fqn": ...}` 字典

### 4.4 回写图库

`scripts/run_mining.py` 将聚合结果通过 Cypher MERGE 写入 Kuzu：

```
python -m scripts.run_mining --sql-dir sql_samples/
```

- JOIN + lineage 聚合 → `MATCH (src:Field), (dst:Field) MERGE (src)-[:FIELD_LINK {source: 'sql_mining'}]->(dst)`
- 表共现聚合 → `MATCH (src:Entity), (dst:Entity) MERGE (src)-[:ENTITY_LINK {source: 'sql_mining'}]->(dst)`

### 4.5 当前范围与限制

| 特性 | 状态 |
| --- | --- |
| JOIN 提取 | 已实现 |

## 5. 机制③ —— 图算法洞察（NetworkX Python）

所有图算法在 Python 中使用 NetworkX 运行，通过 Cypher 从 Kuzu 导出子图到 NetworkX 内存，结果再回写 Kuzu。

### 5.1 算子矩阵

| 分析目标 | NetworkX 算子 | 回写目标 |
| --- | --- | --- |
| 关键字段识别 | `nx.pagerank()` / `nx.betweenness_centrality()` / `nx.degree_centrality()` | `Field.pagerank`, `Field.betweenness`, `Field.degree` |
| 业务模块发现 | `nx.community.louvain_communities()` | `Field.community_id` |
| 强连通分量 | `nx.connected_components()` / `nx.weakly_connected_components()` | `Field.wcc_id` |
| 字段结构相似 | `nx.jaccard_coefficient()` | FIELD_LINK（source='analysis'） |

### 5.2 实现方式

所有算法在 `analysis/` 目录下，通用模式：

```python
# 1. 从 Kuzu 导出 Field 节点 + FIELD_LINK 边到 NetworkX
G = nx.Graph()
rows = client.execute("MATCH (c:Field) RETURN c.fqn AS fqn")
for r in rows:
    G.add_node(r["fqn"])
rows = client.execute("MATCH (c1:Field)-[r:FIELD_LINK]-(c2:Field) RETURN c1.fqn AS src, c2.fqn AS dst")
for r in rows:
    G.add_edge(r["src"], r["dst"])

# 2. 运行算法
pr = nx.pagerank(G)
communities = nx.community.louvain_communities(G)

# 3. 回写 Kuzu
client.execute("BEGIN TRANSACTION")
for fqn, score in pr.items():
    client.execute("MATCH (c:Field {fqn: $fqn}) SET c.pagerank = $score", ...)
client.execute("COMMIT")
```

### 5.3 启动方式

```
python -m scripts.run_analysis --algo centrality      # PageRank + Betweenness
python -m scripts.run_analysis --algo louvain --resolution 1.5
python -m scripts.run_analysis --algo similarity --top-k 10
python -m scripts.run_analysis --algo all              # 全部运行
```

### 5.4 为什么用 NetworkX 而非 GDS

| 对比项 | NetworkX（当前） | GDS（已否决） |
| --- | --- | --- |
| 数据搬运 | 需从 Kuzu 导出子图到 Python 内存 | 无搬运，库内计算 |
| 多粒度 Louvain | 仅输出最终社区 | `includeIntermediateCommunities` 一次输出 |
| 扩展性 | 受限于 Python 内存（百万级以内可接受） | 亿级节点 |
| 依赖 | networkx 纯 Python，零额外基础设施 | 需要 Neo4j Enterprise + GDS 插件 |
| 部署 | 与 Kuzu 同进程，无需独立图库服务 | 需运行 Neo4j 服务 |

当前阶段数据量 ≤ 10 万节点，NetworkX 模式足够。未来若规模膨胀可考虑迁移至 Kuzu 内置算法接口或独立图库。

## 6. Agent / Tool 层（minimal-harness 集成）

### 6.1 工具注册（10 个工具）

| 工具 | 功能 | 底层 |
| --- | --- | --- |
| `schema_search` | 按关键词搜索 Entity / Field | Kuzu Cypher 查询 |
| `subgraph_fetch` | 按 domain/entity 抽取子图 | Kuzu Cypher 查询 |
| `lineage_trace` | 字段级数据血缘追溯（FIELD_LINK BFS） | Kuzu Cypher BFS |
| `join_path_find` | 两 Entity 间最短 JOIN 路径 | Kuzu Cypher 最短路 |
| `sql_executor` | 查询业务库取实例数据 | PostgreSQL 只读连接 |
| `query_schema_data` | 查询所有实体类型 + 中文名 | Kuzu 查询 Entity 表 |
| `query_schema_props` | 查询指定实体类型的属性 | Kuzu HAS_PROPERTY 遍历 |
| `query_schema_rels` | 查询实体间关系模式 | Kuzu rel table 发现 |
| `risk_check` | Orphan FK、跨域引用、缺 PK 规则 | Kuzu 模式匹配 |
| `graph_insight` | PageRank / Louvain / 全图统计 | 读取算法回写属性 |

### 6.2 服务注册（main.py）

```python
service.add_tool(**SCHEMA_SEARCH_TOOL, handler=schema_search_execute)
service.add_tool(**SUBGRAPH_FETCH_TOOL, handler=subgraph_fetch_execute)
service.add_tool(**LINEAGE_TRACE_TOOL, handler=lineage_trace_execute)
service.add_tool(**JOIN_PATH_FIND_TOOL, handler=join_path_find_execute)
service.add_tool(**SQL_EXECUTOR_TOOL, handler=sql_executor_execute)
service.add_tool(**QUERY_SCHEMA_DATA_TOOL, handler=query_schema_data_execute)
service.add_tool(**QUERY_SCHEMA_PROPS_TOOL, handler=query_schema_props_execute)
service.add_tool(**QUERY_SCHEMA_RELS_TOOL, handler=query_schema_rels_execute)
service.add_tool(**RISK_CHECK_INSIGHT_TOOL, handler=risk_check_insight_execute)
service.add_tool(**GRAPH_INSIGHT_TOOL, handler=graph_insight_execute)
```

### 6.3 治理

- `sql_executor` 限制只读，禁 DDL/DML
- 各工具自带参数校验（sanitize identifier 防注入）
- Kuzu 和 PG 客户端连接在服务启动时初始化，关闭时清理

### 6.4 无多 Agent 编排

当前实现仅注册 10 个独立工具，未实现多 Agent 编排或 router agent。Agent 编排层需在 L4 应用层实现（如通过 prompt 让 LLM 自行选择工具链）。

## 8. 关键风险与对策

| 风险 | 对策 |
| --- | --- |
| 动态拼接 SQL / 非标准方言 sqlglot 解析失败 | 失败隔离 + 正则兜底提表名 + parse_error 日志 |
| Excel 注册表版本控制困难 | 手动 Git 管理 `manual_registry.xlsx`；`exported_registry.xlsx` 为只读导出供审计 |
| Kuzu 数据库损坏或版本升级不兼容 | `get_kuzu_client(recreate=True)` 全量重建 |
| NetworkX 内存不足（节点 > 百万） | 当前字段级别 < 10 万；未来可改用 Kuzu 内置算法或分领域投影 |
| 图与注册表不一致 | 图为投影，幂等重建保证一致性；但挖掘产物不回写 Excel |
| 缺少 LLM 语义层导致关系语义薄弱 | 所有关系无 confidence/frequency，Agent 需靠拓扑推理而非语义推理 |
| 多 Agent 编排未实现 | 需 L4 应用层补充 router agent 或 LLM function call 选择工具链 |
